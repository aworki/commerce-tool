# Shoes Team Content Postfill Design

Status: Approved in brainstorming
Date: 2026-03-30

## Summary

Add a team-scoped content-template system for shoe exports so the workflow can optionally fill four workbook fields after a successful shoe export:

- 商品描述 → column `D`
- 关键信息 → column `G`
- SEO标题 → column `T`
- SEO描述 → column `U`

Each team stores four text templates plus a human-entered team description. Templates may include the `{{title}}` placeholder, which is replaced with the exported product title at write time.

The existing Bun/CLI `shoes-transformer` paths remain the single non-interactive workbook-export abstraction and continue returning one JSON result. `shoes-transformer-with-team-content` is not a second export system; it is an interactive postfill wrapper that composes the base export, then optionally prompts the user to apply team content in the same execution.

The base export also assumes `catalog_items.images_json` already contains canonical OSS public URLs from `catalog-ingestion`. Team-content postfill must not introduce its own image handling or alternate export path.

## Goals

- Keep the current `shoes-transformer` export flow intact.
- Add an optional post-export step that fills four workbook fields.
- Store reusable team-specific templates in PostgreSQL.
- Auto-generate the team primary key in the database.
- Collect `team_description` manually from the user when creating a team template.
- Support a single placeholder contract: `{{title}}`.
- Apply the four filled fields only to the first row of each exported product.
- Allow first-use creation of a missing team template during the same flow.
- Keep the direct Bun CLI export path non-interactive and backward-compatible.
- Add a new interactive wrapper skill around `shoes-transformer`.
- Use a stable internal postfill manifest so warning cleanup and row targeting do not depend on re-reading workbook state heuristically.
- Keep v1 postfill semantics per-product, not per-field.

## Non-goals

- Editing or deleting team templates in the first version.
- Supporting per-team workbook column mappings.
- Supporting additional placeholders beyond `{{title}}`.
- Filling the same four fields on continuation SKU rows.
- Replacing the existing export template selection logic.
- Building a general-purpose content-management UI.
- Supporting an arbitrary apply-later standalone workbook-postfill mode in the first version.
- Supporting per-field partial-success accounting in the first version.

## Existing Repository Constraints

Current shoe export behavior already has a clean pipeline:

- `src/skills/shoesTransformer/runShoesTransform.ts` loads catalog items, normalizes them, builds rows, validates warnings, and writes the workbook.
- `src/skills/shoesTransformer/buildWorkbookRows.ts` already distinguishes a product's first row from continuation rows.
- `src/skills/shoesTransformer/normalizeCatalogItemForShoes.ts` currently leaves `D`, `G`, `T`, and `U` blank and reports them as manual-fill warnings.
- `src/db/schema.ts` currently bootstraps the local PostgreSQL schema and is the right place to extend table creation.
- `src/skills/shoesTransformer/shoesTransformerSkill.ts` is currently a thin non-interactive wrapper around `runShoesTransform`.
- The current Bun CLI entrypoints print one JSON result and exit; they do not support an interactive post-export question loop.

The design therefore needs a clear split between non-interactive export code and an interactive orchestration layer that may optionally run postfill after export.

## Data Model

Add a new PostgreSQL table for team content templates.

```text
team_shoes_content_templates
- id                           INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
- team_description             TEXT NOT NULL
- product_description_template TEXT NOT NULL
- key_information_template     TEXT NOT NULL
- seo_title_template           TEXT NOT NULL
- seo_description_template     TEXT NOT NULL
- created_at                   TIMESTAMPTZ NOT NULL DEFAULT NOW()
- updated_at                   TIMESTAMPTZ NOT NULL DEFAULT NOW()
```

### Model rules

- `id` is generated by the database and is the identifier users choose later.
- `team_description` is freeform text entered manually by the user.
- The four template fields are plain text columns, not JSON.
- Templates may contain `{{title}}`, but the placeholder is optional.
- If a template does not contain `{{title}}`, the fixed text is still valid and is written as-is.

Plain columns are preferred over a JSON blob because the set of fields is fixed, small, and directly maps to four workbook destinations.

## Placeholder Contract

The first version supports one placeholder only:

- `{{title}}`

At creation time:

- reject any `{{...}}` placeholder token other than the exact supported token `{{title}}`
- allow plain text with no placeholder at all

At apply time, `{{title}}` is replaced with the current product title.

Example:

```text
Template:  {{title}} 官网同款低帮运动鞋
Result:    Nike Dunk Low 官网同款低帮运动鞋
```

No other placeholders, conditional syntax, or template functions are supported in this version.

## Workbook Mapping Contract

The postfill step writes exactly four columns on the exported `商品信息` sheet:

- 商品描述 → `D`
- 关键信息 → `G`
- SEO标题 → `T`
- SEO描述 → `U`

### Row-level behavior

- Write these fields only on the first row for each exported product.
- Leave continuation rows unchanged.
- Preserve the existing workbook layout, formulas, styles, and all non-target columns.

This matches the current export structure, where shared product-level content belongs on the first row and later rows only carry SKU-specific data.

## Execution Modes

There are two explicit execution modes.

### Non-interactive mode

These existing entrypoints remain non-interactive:

- `bun run transform:shoes`
- `bun run skill:shoes-transformer`
- `createShoesTransformerSkill()` / `shoes-transformer`

They:

- run the base shoe export only
- print or return one JSON result
- exit without prompting
- keep the current warning contract, including the manual-fill warnings for `D/G/T/U`

### Interactive mode

Add a new wrapper skill named `shoes-transformer-with-team-content`.

It:

- accepts the same selector and output inputs as `shoes-transformer`
- runs the export and optional postfill inside one interactive execution
- keeps the export manifest in memory inside the same execution boundary
- asks whether to fill 商品描述 / 关键信息 / SEO标题 / SEO描述 after a successful export
- optionally performs team-template selection and workbook postfill
- returns a final user-facing result that reflects the postfill outcome

The workbook postfill module is internal to this wrapper flow and is not a generic apply-later tool in v1.

## Internal Export Artifact Contract

Add an internal-only export artifact type, for example `ShoesTransformExecution`, containing at least:

- `publicResult: ShoesTransformResult`
- `manifest: ShoesPostfillManifestEntry[]`

Each `ShoesPostfillManifestEntry` contains at least:

- `sourceId`
- `title`
- `firstRowNumber`

The existing public JSON contract remains `ShoesTransformResult`. The manifest is available only inside the wrapper flow and is not printed by existing CLI commands.

The base export logic should therefore expose one internal function that produces `ShoesTransformExecution`, while the existing public helpers continue returning only `ShoesTransformResult`.

## User Workflow

### Primary path

1. The assistant runs the new `shoes-transformer-with-team-content` wrapper skill.
2. The wrapper skill runs the existing export logic and captures the internal export artifact.
3. If the export fails, the workflow returns that failure and stops.
4. If the export succeeds, the wrapper skill asks whether to fill 商品描述 / 关键信息 / SEO标题 / SEO描述.
5. If the user declines, the workflow ends with the workbook untouched beyond the base export.
6. If the user accepts, the wrapper flow enters the team-template selection path.
7. After postfill, the workflow returns a final summary that includes the base export result plus postfill status.

### Final interactive result shape

The interactive path returns one final structured summary containing:

- the base export result
- whether postfill was skipped or applied
- the selected team `id` when applied
- how many products were updated
- postfill warnings/errors, if any
- the effective final warning set shown to the user

When postfill succeeds, the final warning set should no longer include the original manual-fill warnings for columns `D/G/T/U` for the products that were updated successfully.

When postfill is skipped or fails before any workbook update, those original warnings remain visible.

### Team-template selection path

The user chooses one of two actions:

- use an existing team template
- create a new team template

#### Use existing team template

1. List existing teams with:
   - `id`
   - `team_description`
2. User selects a team by `id`.
3. Load that team's four templates.
4. Replace `{{title}}` per exported product.
5. Write the generated values into columns `D/G/T/U` on the product's first row.
6. Return a summary of which team was used and how many products were updated.

#### Create a new team template

1. Prompt for:
   - `team_description`
   - 商品描述模板
   - 关键信息模板
   - SEO标题模板
   - SEO描述模板
2. Trim leading/trailing whitespace on all five inputs.
3. Reject the creation if `team_description` is empty after trimming.
4. Reject the creation if any of the four template inputs are empty after trimming.
5. Reject the creation if any `{{...}}` token appears other than `{{title}}`.
6. Insert the template row into PostgreSQL.
7. Return the generated `id`.
8. Immediately apply the new template to the just-exported workbook.
9. Return the same update summary.

### Empty-state behavior

If the user asks to fill the four fields and no team templates exist yet, the flow should skip the selection list and go directly to new-team creation.

## System Structure

### Database layer

Add a new module:

- `src/db/teamShoesTemplates.ts`

Responsibilities:

- create a team template
- list available team templates
- fetch a team template by `id`

This module should contain only typed database access and not workbook or prompting logic.

### Export execution layer

Add or expose one internal export execution function, for example `runShoesTransformExecution(...)`, that returns `ShoesTransformExecution`.

Responsibilities:

- run the existing export flow
- preserve the current public `ShoesTransformResult`
- build the internal postfill manifest with `sourceId`, `title`, and `firstRowNumber`

The existing public `runShoesTransform(...)` can call this internal function and continue returning only `ShoesTransformResult`.

### Postfill business layer

Add a dedicated workbook postfill module under `src/skills/shoesTransformer/`.

Suggested responsibilities:

- generate field values from a team template and product title
- open the exported workbook
- locate each product's first row using the postfill manifest
- write values to `D/G/T/U`
- save the workbook and return a structured summary

### Prompt/orchestration layer

Add a separate orchestration module for the wrapper skill that handles:

- ask whether to fill the four fields
- list existing teams
- collect new-team input when needed
- validate template inputs
- invoke the workbook postfill module

The prompting/orchestration layer should call the DB module, internal export execution layer, and workbook module, while the lower layers remain non-interactive.

## Entry Points

### Base export entrypoints

The current export flow in `src/skills/shoesTransformer/runShoesTransform.ts`, `src/skills/shoesTransformer/shoesTransformerSkill.ts`, `src/cli/transformShoes.ts`, and `src/cli/shoesTransformerSkill.ts` remain responsible for:

- loading products from the DB
- normalizing export data
- building rows
- writing the base workbook
- returning one non-interactive JSON result

They should not directly prompt the user.

### New wrapper skill entrypoint

Add a new skill named `shoes-transformer-with-team-content`.

Accepted inputs:

- the same selector inputs already accepted by `shoes-transformer`
- `--output <output.xlsx>`
- optional `--template <template.xlsx>` and `--tags ...`

Rejected inputs:

- arbitrary workbook paths for apply-later use
- requests that bypass the wrapper and still expect interactive postfill from the existing Bun CLI commands

The wrapper skill then:

1. calls the internal export execution function
2. asks whether to apply team content
3. handles existing-team selection or new-team creation
4. performs workbook postfill when requested
5. returns one final interactive summary

This keeps the core export contract stable while still allowing the user to continue into the optional postfill step immediately after export.

## Apply Algorithm

For each exported product in the postfill manifest:

1. load the selected team template
2. generate four output strings by replacing `{{title}}`
3. trim the generated values
4. if all four values remain non-empty, write the generated values to `D`, `G`, `T`, and `U` on that product's first row
5. if any generated value is empty, treat that product as not updated and leave all four target cells untouched
6. leave all continuation rows unchanged

The algorithm should operate directly on the workbook that the base export just created rather than re-running the export logic.

### Postfill manifest contract

The wrapper flow must retain an internal manifest for each exported product containing at least:

- `sourceId`
- `title`
- `firstRowNumber`

This manifest is not required in the existing public JSON contract for non-interactive CLI output. It exists so the postfill step and final warning reconciliation can target products deterministically without inferring identity from workbook contents.

### Row-location contract

The postfill logic should rely on the current workbook contract established by the exporter:

- sheet name: `商品信息`
- data rows begin at row `5`
- product-level shared content is present on the product's first row
- continuation rows for the same product do not repeat shared content in column `B`

The postfill step should therefore use the manifest's `firstRowNumber` as the primary targeting key and only write the four fields on rows that represent product first rows.

## Warning Contract

The base export produces warnings before any optional postfill runs. The wrapper skill is responsible for reconciling those warnings with the final workbook state.

Rules:

- if postfill is skipped, return the original warning set unchanged
- if a product postfill succeeds, remove that product's manual-fill warnings for `D`, `G`, `T`, and `U` from the final warning set
- if a product postfill does not succeed, retain all four original manual-fill warnings for that product
- keep unrelated export warnings unchanged
- report postfill-specific failures in a separate postfill warning/error section as well as in the final summary

## Error Handling

### Hard-stop errors

Stop the postfill step immediately when:

- the workbook cannot be opened
- the `商品信息` sheet is missing
- the selected team template cannot be loaded after selection
- saving the workbook fails globally

On a hard-stop error, no manual-fill warnings are removed from the base export result because the workbook cannot be trusted to have changed.

### Per-product errors

Continue processing other products and report structured warnings when:

- a target row cannot be located for a specific product
- any of the four generated values for a product is empty after trimming

For a per-product error, that product remains fully unchanged for columns `D/G/T/U`, and all four of its original manual-fill warnings remain in the final warning set.

### Validation policy

Keep first-version validation intentionally light:

- do not require `{{title}}` to appear
- do reject empty or whitespace-only `team_description`
- do reject empty or whitespace-only template inputs at creation time
- do reject unsupported `{{...}}` placeholder tokens other than `{{title}}`
- after placeholder replacement, trim generated values before writing
- if any of the four generated values is empty after trimming, treat that product as not updated
- do not add speculative rules for maximum length or banned characters unless workbook behavior proves they are necessary

## Testing Strategy

Add focused tests for the new behavior.

### Database tests

- create a team template
- list existing team templates
- fetch a team template by id

### Template-generation tests

- replace `{{title}}` correctly
- preserve fixed text when no placeholder exists
- trim inputs and reject empty `team_description`
- trim inputs and reject empty template creation
- reject unsupported `{{...}}` tokens
- generate all four field outputs independently

### Workbook postfill tests

- write `D/G/T/U` on the first row only
- leave continuation rows unchanged
- process multiple products in order using manifest row numbers
- preserve unrelated cell contents
- when a product fails validation or row targeting, leave all four target cells unchanged for that product

### Workflow tests

- non-interactive CLI export returns once and never prompts
- wrapper skill declines postfill and leaves export output unchanged
- wrapper skill creates a new team template and applies it immediately
- wrapper skill selects an existing team template and applies it successfully
- empty-state selection routes directly into new-team creation
- final warning set drops successful `D/G/T/U` manual-fill warnings after postfill
- final warning set keeps all four manual-fill warnings for products whose postfill did not succeed
- workbook-open failure keeps the original warning set unchanged
- missing `商品信息` sheet keeps the original warning set unchanged
- workbook save failure keeps the original warning set unchanged

## Rollout Notes

This feature should be introduced as an additive workflow. Users who do not want the four fields filled can continue using `shoes-transformer` exactly as before by declining the prompt.

The first version intentionally optimizes for getting the workflow live quickly:

- fixed four-field schema
- fixed workbook column mapping
- create/select only, no template editing
- one placeholder contract
- no arbitrary apply-later standalone mode
- per-product all-or-nothing postfill semantics
- wrapper-skill orchestration with an internal export manifest

If future teams need richer content management, editing support can be added later without changing the workbook contract established here.
